# 需求文档

## 简介

本文档规定了为 CoCreateHub 协作文档编辑平台实现企业级文档版本历史管理功能的需求。该功能使用户能够跟踪、查看、比较和恢复文档的先前版本，并通过智能快照策略优化存储使用。

## 术语表

- **版本历史系统（Version_History_System）**: 完整的文档版本管理子系统
- **版本快照（Version_Snapshot）**: 特定时间点保存的文档状态，以 Yjs CRDT 二进制数据存储
- **版本时间线（Version_Timeline）**: 按时间顺序显示文档所有版本快照的界面
- **版本对比视图（Version_Diff_View）**: 显示两个文档版本之间差异的可视化比较界面
- **智能快照策略（Smart_Snapshot_Strategy）**: 基于手动保存、重大更改和时间合并来决定何时创建版本快照的智能算法
- **版本卡片（Version_Card）**: 显示单个版本元数据（编辑者、时间戳、操作类型）的 UI 组件
- **版本菜单（Version_Menu）**: 文件列表中每个文件的三点下拉菜单
- **自动保存（Auto_Save）**: 系统自动创建的版本快照
- **自动里程碑（Auto_Milestone）**: 检测到重大文档更改时自动创建的版本快照
- **定时合并（Timed_Merge）**: 在时间窗口内合并频繁版本快照的整合过程
- **版本锁定（Version_Lock）**: 防止特定版本被删除或修改的保护机制
- **版本标签（Version_Label）**: 用户为版本附加的自定义文本标记，便于识别
- **版本预览（Version_Preview）**: 特定版本文档内容的只读视图
- **版本恢复（Version_Restore）**: 用先前版本的内容替换当前文档内容的操作
- **版本导出（Version_Export）**: 将特定版本下载为独立文件的操作
- **差异统计（Diff_Statistics）**: 显示版本之间更改程度的定量指标（新增、删除、修改）

## 需求

### 需求 1：版本历史访问

**用户故事：** 作为文档协作者，我希望从文件菜单访问版本历史，以便在不离开工作区的情况下查看和管理文档版本。

#### 验收标准

1. 当用户点击文件列表中文件的三点菜单时，版本菜单应显示"版本历史"选项
2. 当用户选择"版本历史"选项时，版本历史系统应打开版本时间线界面
3. 版本时间线界面应使用深色主题和毛玻璃背景效果
4. 版本时间线界面应按时间倒序（最新的在前）显示所有版本快照

### 需求 2：智能版本快照创建

**用户故事：** 作为系统管理员，我希望有智能版本快照策略，以便在保留重要文档状态的同时优化存储使用。

#### 验收标准

1. 当智能快照策略检测到超过阈值的重大文档更改时，版本历史系统应创建自动里程碑版本快照
2. 当在可配置的时间窗口（默认 5 分钟）内创建多个版本快照时，版本历史系统应执行定时合并将它们整合为单个快照
3. 版本历史系统在定时合并期间应保留每个时间窗口内最新的快照
4. 版本历史系统在定时合并期间不应合并锁定的快照

### 需求 3：版本时间线显示

**用户故事：** 作为文档协作者，我希望看到所有文档版本的时间线，以便了解文档随时间的演变。

#### 验收标准

1. 对于时间线中的每个版本快照，版本时间线应显示一个版本卡片
2. 版本卡片应显示编辑者的用户名
3. 版本卡片应以人类可读的格式显示创建时间戳
4. 版本卡片应显示快照类型（自动里程碑或标准保存）
5. 版本卡片应显示任何附加的版本标签
6. 当版本启用了版本锁定时，版本卡片应显示锁定图标

### 需求 4：版本预览

**用户故事：** 作为文档协作者，我希望预览先前的版本，以便在不修改当前版本的情况下查看文档内容。

#### 验收标准

1. 当用户点击版本卡片时，版本历史系统应打开版本预览界面
2. 版本预览应显示该版本快照时存在的文档内容
3. 版本预览应为只读并防止编辑
4. 版本预览应将 Yjs CRDT 二进制数据渲染为 Slate 编辑器格式
5. 版本预览应在标题中显示版本元数据（编辑者、时间戳、标签）

### 需求 5：版本比较

**用户故事：** 作为文档协作者，我希望比较两个版本，以便识别它们之间的更改。

#### 验收标准

1. 当用户选择两个版本卡片并点击"比较"时，版本历史系统应打开版本对比视图
2. 版本对比视图应并排显示两个版本
3. 版本对比视图应以绿色突出显示文本新增
4. 版本对比视图应以红色突出显示文本删除
5. 版本对比视图应以黄色突出显示文本修改
6. 版本对比视图应显示差异统计，显示新增、删除和修改的计数

### 需求 6：版本恢复

**用户故事：** 作为文档协作者，我希望恢复先前的版本，以便撤销不需要的更改。

#### 验收标准

1. 当用户点击版本卡片上的"恢复"按钮时，版本历史系统应显示确认对话框
2. 当用户确认恢复时，版本历史系统应用所选版本快照的内容替换当前文档内容
3. 当恢复完成时，版本历史系统应创建一个新的版本快照标记恢复操作
4. 当恢复完成时，版本历史系统应更新文档的 updatetime 时间戳
5. 当恢复完成时，版本历史系统应通过 Yjs 向所有连接的协作者广播更改

### 需求 7：版本标签

**用户故事：** 作为文档协作者，我希望为版本添加标签，以便标记重要的里程碑。

#### 验收标准

1. 当用户点击版本卡片上的"添加标签"按钮时，版本历史系统应显示文本输入对话框
2. 当用户输入标签文本并确认时，版本历史系统应将版本标签附加到该版本快照
3. 版本历史系统应在数据库中存储与版本 ID 关联的版本标签
4. 版本历史系统应在相应的版本卡片上显示所有版本标签
5. 当用户点击版本标签时，版本历史系统应允许编辑或删除标签

### 需求 8：版本搜索

**用户故事：** 作为文档协作者，我希望在版本历史中搜索，以便快速找到特定版本。

#### 验收标准

1. 版本时间线应在顶部显示搜索输入框
2. 当用户在搜索框中输入文本时，版本历史系统应按编辑者用户名、版本标签或时间戳过滤版本卡片
3. 版本历史系统应在用户输入时实时更新显示的版本卡片
4. 当搜索框为空时，版本历史系统应显示所有版本快照

### 需求 9：版本导出

**用户故事：** 作为文档协作者，我希望导出特定版本，以便将它们保存为独立文件。

#### 验收标准

1. 当用户点击版本卡片上的"导出"按钮时，版本历史系统应显示导出格式选项
2. 当用户选择格式并确认时，版本历史系统应将版本快照内容转换为所选格式
3. 版本历史系统应支持导出为纯文本（.txt）格式
4. 版本历史系统应支持导出为 Markdown（.md）格式
5. 版本历史系统应触发浏览器下载，文件名格式为："{docname}_v{vid}_{timestamp}.{extension}"

### 需求 10：版本锁定

**用户故事：** 作为文档协作者，我希望锁定重要版本，以便保护它们免受自动删除或合并。

#### 验收标准

1. 当用户点击版本卡片上的"锁定"按钮时，版本历史系统应为该版本快照启用版本锁定
2. 当启用版本锁定时，版本历史系统应在版本卡片上显示锁定图标
3. 当启用版本锁定时，智能快照策略应将该版本从定时合并操作中排除
4. 当用户点击锁定版本上的锁定图标时，版本历史系统应禁用版本锁定
5. 版本历史系统应在数据库中存储锁定状态

### 需求 11：版本历史 UI 样式

**用户故事：** 作为文档协作者，我希望有美观的版本历史界面，以便该功能使用起来令人愉悦。

#### 验收标准

1. 版本时间线界面应使用与应用程序主题一致的深色配色方案
2. 版本时间线界面应应用毛玻璃（backdrop-filter: blur）背景效果
3. 版本卡片组件应具有圆角和细微的阴影效果
4. 当用户的光标悬停在版本卡片组件上时，应显示悬停效果
5. 版本时间线在版本之间导航时应使用平滑的滚动动画

### 需求 12：版本历史性能

**用户故事：** 作为文档协作者，我希望版本历史快速加载，以便无延迟地访问版本。

#### 验收标准

1. 当用户打开版本时间线时，版本历史系统应在 500ms 内加载并显示前 20 个版本快照
2. 当用户滚动到版本时间线底部时，版本历史系统应加载接下来的 20 个版本快照（无限滚动分页）
3. 版本历史系统应在浏览器内存中缓存已加载的版本快照元数据
4. 当加载版本预览内容时，如果操作超过 200ms，版本历史系统应显示加载指示器

### 需求 13：版本快照存储优化

**用户故事：** 作为系统管理员，我希望优化版本存储，以便数据库大小保持可管理。

#### 验收标准

1. 版本历史系统应将版本快照内容存储为压缩的 Yjs CRDT 二进制数据（LONGBLOB）
2. 当智能快照策略执行定时合并时，版本历史系统应从数据库中删除合并的快照
3. 版本历史系统应维护每个文档的可配置最大版本数（默认 100 个版本）
4. 当版本数超过最大值时，版本历史系统应删除最旧的未锁定、未标记的版本
5. 版本历史系统应记录所有版本删除以供审计

### 需求 14：版本历史 API 端点

**用户故事：** 作为前端开发人员，我希望有用于版本操作的 RESTful API 端点，以便实现版本历史 UI。

#### 验收标准

1. 版本历史系统应提供 GET 端点 `/api/doc/{docid}/versions` 返回分页的版本快照元数据
2. 版本历史系统应提供 GET 端点 `/api/doc/version/{vid}` 返回完整的版本快照内容
3. 版本历史系统应提供 POST 端点 `/api/doc/{docid}/version/restore` 接受版本 ID 进行恢复
4. 版本历史系统应提供 POST 端点 `/api/doc/version/{vid}/label` 接受标签文本进行添加或更新
5. 版本历史系统应提供 POST 端点 `/api/doc/version/{vid}/lock` 切换版本锁定状态
6. 版本历史系统应提供 GET 端点 `/api/doc/version/{vid}/export` 接受格式参数并返回文件内容

### 需求 15：版本历史错误处理

**用户故事：** 作为文档协作者，我希望在版本操作失败时有清晰的错误消息，以便了解出了什么问题。

#### 验收标准

1. 当版本操作因网络问题失败时，版本历史系统应显示用户友好的中文错误消息
2. 当用户尝试恢复不再存在的版本时，版本历史系统应显示"版本不存在"
3. 当用户尝试访问已删除文档的版本历史时，版本历史系统应显示"文档已删除"
4. 当版本预览加载内容失败时，版本历史系统应显示"加载失败，请重试"
5. 版本历史系统应将所有错误记录到后端以供调试

### 需求 16：协作版本感知

**用户故事：** 作为文档协作者，我希望看到其他用户何时查看版本历史，以便了解并发的版本操作。

#### 验收标准

1. 当用户打开版本时间线时，版本历史系统应向所有连接的协作者广播"viewing_history"事件
2. 当另一个用户正在查看版本历史时，版本历史系统应显示指示器显示其用户名
3. 当用户恢复版本时，版本历史系统应向所有连接的协作者广播通知
4. 通知应显示恢复用户的用户名和正在恢复的版本时间戳
